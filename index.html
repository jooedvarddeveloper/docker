😤

A fejlesztő gépe

A fejlesztő megírja az alkalmazást, leteszteli a saját gépén → minden tökéletesen működik.

Python 3.11, specifikus library verziók, lokális env változók – mind pontosan beállítva.

💥

A szerver / kollégák gépe

Feltelepítik egy másik gépre → semmi nem működik.

Más Python verzió, hiányzó library, eltérő OS, más env konfiguráció – a környezet nem reprodukálható.

Ezt a problémát hívják "works on my machine" szindrómának – és évtizedek óta gyötri a szoftveripart. A gyökér oka: az alkalmazás és a futtatási környezete szét van választva. A Docker megoldása: csomagold össze az alkalmazást a teljes futtatási környezetével együtt – OS library-k, runtime, konfig, minden – egyetlen hordozható egységbe.

Mi az a Docker?
Definíció és alapkoncepció
📦

Konténerizációs platform

A Docker egy nyílt forráskódú eszköz, amely lehetővé teszi alkalmazások konténerekbe csomagolását.

Egy konténer tartalmazza az alkalmazást és az összes függőségét: runtime, system library-k, konfigurációs fájlok – mindent, ami a futáshoz kell.

Az így becsomagolt alkalmazás bármely Docker-kompatibilis gépen identikusan fut.

📅

Rövid történet

2013 – Solomon Hykes publikálta a Docker-t (dotCloud startup)
2014 – Docker Hub elindult, robbanásszerű növekedés
2015 – OCI (Open Container Initiative) szabvány alapítása
2017 – Moby projekt: a Docker engine open source
Ma – 14M+ fejlesztő, 13 milliárd pull/hó

🔑

A konténer lényege

A konténer egy process-szintű izolációs egység. Saját fájlrendszert, hálózatot és process-teret lát – de a host OS kernelét osztja meg.

Ez teszi sokkal könnyebbé és gyorsabbá a virtuális gépeknél: nincs szükség teljes guest OS-re.

⚙️

Mögöttes Linux technológiák

A Docker a Linux kernel két funkciójára épül:

Namespaces – izolálják a konténer process-eit, hálózatát, fájlrendszerét
cgroups – korlátozzák a CPU, memória és I/O erőforrás-használatot

Konténer vs. Virtuális Gép
Mindkettő izolációt nyújt – de nagyon különböző módon
🖥️

Virtuális gép (VM)

A hypervisor (pl. VMware, VirtualBox, KVM) teljes hardvert szimulál.

Minden VM saját guest OS-t futtat (kernel + userspace), saját dedikált memóriával és CPU-val.

Eredmény: erős izoláció, de nagy overhead – GB-os méret, perces indulás.

📦

Docker konténer

Nincs hypervisor, nincs guest OS. A konténerek közvetlenül a host kernel felett futnak.

Az izoláció namespaces + cgroups segítségével valósul meg – OS-szintű, nem hardver-szintű.

Eredmény: közel natív teljesítmény, MB-os méret, másodperces indulás.

📋
DOCKERFILE

Build instrukciók

Egy szöveges fájl, amely lépésről lépésre leírja, hogyan kell felépíteni az image-t.

Tartalmazza: melyik base image-ből induljunk, milyen csomagokat telepítsünk, milyen fájlokat másoljunk be, hogyan induljon el az alkalmazás.

🖼️
IMAGE

Read-only snapshot

A docker build parancs a Dockerfile alapján létrehoz egy immutable (nem módosítható) rétegelt image-t.

Az image tartalmazza a teljes fájlrendszert, amit a konténer futás közben használ. Megosztható registry-n keresztül.

⚙️
CONTAINER

Futó process-izolátum

A docker run parancs elindít egy konténert az image-ből. A konténer az image tetejére helyez egy írható réteget (copy-on-write).

Egy image-ből egyszerre több konténer is futtatható, teljesen függetlenül.

A kapcsolat:   Dockerfile → build → Image → push/pull → Registry → run → Container (futó process)   — Ez a Docker teljes életciklusa.

⚡

Build cache – miért fontos?

Docker minden réteget cachelel. Ha egy réteg bemenete nem változott, újrahasználja a cache-t – nem futtatja újra azt a lépést.

Ezért kritikus a sorrend: a ritkán változó dolgok (pl. csomagok telepítése) kerüljenek előre, a sűrűn változó kód hátra.

🗂️

OverlayFS – Union fájlrendszer

A Docker alapértelmezett tárolóillesztője. Több read-only réteget és egy írható réteget egyetlen, egységes fájlrendszerként jelenít meg a konténernek.

Copy-on-Write (CoW): olvasás az alsó rétegből, íráskor a felső rétegbe másolja az elemet.

A Dockerfile – sorról sorra
Minden utasítás mit csinál pontosan?
# Alap image: Node.js 20 + minimális Alpine Linux
FROM node:20-alpine
# Munkakönyvtár beállítása a konténerben
WORKDIR /app
# ELŐBB a dependency lista – cache optimalizáció!
COPY package*.json ./
RUN npm ci --only=production
# Forráskód másolása (ez változik a leggyakrabban)
COPY . .
# Dokumentálja, hogy a konténer ezen a porton hallgat
EXPOSE 3000
# Ne root-ként fusson az alkalmazás!
USER node
# A konténer indulásakor futtatott parancs
CMD ["node", "src/index.js"]
🔍

CMD vs. ENTRYPOINT

CMD – default parancs, felülírható a docker run-nál

ENTRYPOINT – rögzített belépési pont; a CMD ilyenkor argumentum

Best practice: ENTRYPOINT a végrehajtható, CMD a default argument

🏗️

Multi-stage build

Több FROM utasítás egy Dockerfile-ban. Az első stage-ben buildelünk (compiler, test tools), a másodikba csak a futáshoz szükséges artifactokat másoljuk át.

Eredmény: a prod image nem tartalmazza a build toolchaint → kisebb, biztonságosabb.

Docker Networking
Hogyan kommunikálnak a konténerek egymással és a külvilággal?
🌉

bridge (default)

Docker virtuális switch-et hoz létre. Az ugyanazon a bridge hálózaton lévő konténerek konténernév alapján elérik egymást (beépített DNS).

docker network create mynet
Fejlesztői környezethez ideális.

🖥️

host

A konténer a host hálózati interfészeit közvetlenül használja – nincs NAT, nincs virtualizáció.

Nagy hálózati teljesítmény, de nincs hálózati izoláció.

Csak Linux hoston működik.

🚫

none

Teljes hálózati izoláció – a konténernek nincs hálózati interfésze.

Egyszeri futású batch jobokhoz, ahol biztonsági okokból nem szabad hálózati hozzáférés.

Port mapping (-p): a host és konténer portját köti össze. -p 8080:80 → a host 8080-as portjára érkező forgalom a konténer 80-as portjára kerül (NAT/iptables szabályok via Docker daemon).

Custom network DNS: user-defined bridge hálózaton a konténer neve automatikusan feloldható hostnévként → http://db:5432 eléri a db nevű konténert.

Volumes és adatperzisztencia
A konténer ephemeral (ideiglenes) – az adatok nem feltétlenül azok
⚠️

A konténer ideiglenes természete

A konténer írható rétege a konténerrel együtt törlődik. Ha az alkalmazás az adatokat a konténer fájlrendszerébe írja, azok elvesznek a konténer törlésekor.

Adatbázisoknál, feltöltött fájloknál, logfájloknál ez komoly probléma lenne – erre való a volume.

💾

Named Volume

A Docker engine kezeli a tárhelyet (/var/lib/docker/volumes/). A konténertől független életciklusa van – konténer törlésével nem törlődik.

docker volume create pgdata
docker run -v pgdata:/var/lib/postgresql/data postgres

📂

Bind Mount

A host fájlrendszerének egy könyvtárát mountolja a konténerbe. Közvetlen hozzáférés a host fájljaihoz – fejlesztéskor ideális, mert a kód módosítása azonnal látszik a konténerben.

docker run -v $(pwd):/app node

⚡

tmpfs Mount

Kizárólag a host memóriájában él – nem kerül lemezre. Érzékeny, ideiglenes adatokhoz (pl. session tokenek, titkosítási kulcsok).

docker run --tmpfs /tmp:rw,size=100m app

Docker Compose
Multi-container alkalmazások deklaratív definíciója
services:
api: # saját alkalmazásunk
build: .
ports: ["3000:3000"]
environment:
DATABASE_URL: postgres://user:pass@db/mydb
REDIS_URL: redis://cache:6379
depends_on:
db: { condition: service_healthy }
networks: [backend]
db: # PostgreSQL
image: postgres:16-alpine
environment:
POSTGRES_PASSWORD: secret
volumes: [pgdata:/var/lib/postgresql/data]
healthcheck:
test: ["CMD", "pg_isready"]
interval: 5s
networks: [backend]
cache: # Redis
image: redis:7-alpine
networks: [backend]
volumes:
pgdata:
networks:
backend:
📋

Miért Compose?

Kézzel minden konténert külön docker run-nal indítani és összekapcsolni rendkívül körülményes.

A Compose egyetlen YAML fájlban deklarálja az összes service-t, hálózatot és volume-ot → egy paranccsal elindítható, leállítható az egész stack.

docker compose up -d # stack indítása
docker compose up --build # újrabuilddel
docker compose logs -f api # service logok
docker compose ps # service státuszok
docker compose down -v # leállítás + volume
docker compose exec api sh # shell a service-be
Docker Biztonsági Alapelvek
Alapértelmezetten a Docker nem biztonságos – te kell, hogy az legyen
👤

Principle of Least Privilege – ne root!

Alapértelmezetten a konténer root-ként fut. Ha az alkalmazást kompromittálják, a támadó root jogosultságot kap a konténerben – és privilege escalation révén esetleg a hoston is.

USER node a Dockerfile végén – mindig.

🤫

Secrets kezelése – soha ne image-be!

Ha egy secret bekerül egy image rétegbe, a docker history paranccsal kiolvasható, még ha törölted is.

Megoldás: .env fájl (ne commitold!), Docker Secrets, Vault, vagy a cloud provider secret managere (AWS SSM, GCP Secret Manager).

🏔️

Minimális base image

alpine (5 MB) és distroless képek nem tartalmaznak shell-t, csomagkezelőt, debug toolokat.

Kisebb image = kisebb attack surface. A CVE scannereknek is kevesebb a dolga.

🔍

Image vulnerability scanning

Trivy, Snyk, Docker Scout – integrálhatók CI/CD pipeline-ba.

Pineld a base image-et digest alapján: node@sha256:abc123… → garantált reprodukálhatóság, nincs titkos frissítés.

🔒

Read-only fájlrendszer

--read-only flag: a konténer nem tud a saját fájlrendszerébe írni.

Kombináld --tmpfs /tmp-vel az ideiglenes írási igényekhez → a futásidejű módosítások teljes mértékben kontrolláltak.

📊

Resource limits

Limit nélkül egy konténer felzabálhatja a host összes erőforrását (noisy neighbor problem).

--memory 512m  |  --cpus 0.5
Compose-ban: deploy.resources.limits

Docker a CI/CD Pipeline-ban
GitHub Actions – build, scan, push, deploy automatizálva
name: Docker CI/CD
on: { push: { branches: [main] } }
jobs:
build-push:
runs-on: ubuntu-latest
steps:
- uses: actions/checkout@v4
- name: Setup Buildx (multi-platform support)
uses: docker/setup-buildx-action@v3
- name: Login to Docker Hub
uses: docker/login-action@v3
with: { username: ${{ secrets.DOCKERHUB_USER }}, password: ${{ secrets.DOCKERHUB_TOKEN }} }
- name: Build & Push (amd64 + arm64)
uses: docker/build-push-action@v5
with:
push: true
tags: user/myapp:${{ github.sha }},user/myapp:latest
platforms: linux/amd64,linux/arm64
cache-from: type=gha # GitHub Actions cache
cache-to: type=gha,mode=max
- name: Security Scan – Trivy
uses: aquasecurity/trivy-action@master
with: { image-ref: 'user/myapp:latest', severity: 'HIGH,CRITICAL', exit-code: '1' }

💡 Immutable image tag-ek: a ${{ github.sha }} biztosítja, hogy minden Git commit pontosan egy image-hez rendelhető → teljes auditálhatóság, egyszerű rollback. A latest tag csak kényelmi hivatkozás – prodban mindig konkrét SHA vagy verzió tag-et használj!
